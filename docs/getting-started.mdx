# Getting Started

OneGlanse supports three setup paths.

## 1. Use the Cloud App

Use the hosted app if you want the fastest setup:

- App: `https://app.oneglanse.com`
- Landing: `https://oneglanse.com`
- Docs: `https://docs.oneglanse.com/`

Recommended flow:

1. Sign in
2. Create or select a workspace
3. Add prompts
4. Review responses, sources, and dashboard metrics

## 2. Run It Locally

Use local mode when you want to run the web app and agent on your own machine.

### Requirements

- Node.js 20+
- pnpm 10+
- Docker + Docker Compose

### Install

```bash
pnpm install
```

### Start

```bash
pnpm local
```

This:

- Creates `.env` and `apps/agent/.env` if missing
- Starts Postgres, ClickHouse, and Redis
- Runs migrations
- Starts the web app and the agent
- Opens the app at `http://localhost:3000`

### Local Notes

- Local mode supports interactive provider auth
- Local mode supports manual prompt runs
- Local mode does not support recurring scheduling

## 3. Self-Host It

Use self-hosted mode when you want the full product in your own infrastructure.

### Install

```bash
pnpm install
```

### Start

```bash
pnpm self-host
```

This starts:

- The main app stack
- The public stack
- It tries to pull the latest published images before starting the services
- If pull access is unavailable, it falls back automatically to a local Docker build

Default ports:

- Landing: `3000`
- App: `3001`
- Auth upload API: `3333`

Persistent self-hosted state defaults to `/opt/oneglanse/storage`. Override it
with `ONEGLANSE_STORAGE_ROOT` if your VPS should store data elsewhere.

Use `pnpm self-host:build` only if you intentionally want to build the Docker
images from local source instead of pulling the published images.

### Provider Auth Setup

Interactive provider login (Google, LinkedIn, etc.) cannot run on a headless server. You capture sessions locally and transfer them to the VPS.

**Step 1 — Start your VPS first**

```bash
pnpm self-host
```

The auth upload API must be running on port `3333` before you transfer sessions.

**Step 2 — Capture and transfer sessions from your local machine**

```bash
pnpm auth -- --upload-url http://YOUR_VPS_HOST:3333/auth/sessions --upload-token YOUR_TOKEN
```

Replace `YOUR_VPS_HOST` with your server's IP or hostname, and `YOUR_TOKEN` with the value of `AGENT_AUTH_UPLOAD_TOKEN` in your VPS `.env` file.

This command:

1. Opens a browser window for each provider
2. Lets you log in interactively
3. Captures the session state (cookies + localStorage)
4. Compresses and uploads it to the VPS over HTTP

**Step 3 — Verify transfer**

The agent reads the transferred sessions at the start of each job run. If the session hash has changed since the last run, it reseeds the browser profile automatically before executing prompts.

### Self-Hosted Notes

- Interactive provider login does not run on the server — use the auth transfer command above
- Re-run the transfer command any time a provider session expires
- Recurring scheduling is supported

## Choosing the Right Mode

- Use Cloud for the fastest product access
- Use Local for development and manual runs
- Use Self-Hosted for production control and private infrastructure
