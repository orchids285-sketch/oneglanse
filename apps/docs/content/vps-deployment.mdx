# VPS Deployment

This guide reflects the current self-hosted runtime in this repository.

## What Runs on the VPS

`pnpm self-host` starts two separate stacks.

### App stack

- `web`
- `agent-worker`
- `redis`
- `db` (Postgres)
- `clickhouse`
- `migrate`
- auth upload API on port `3333`

### Public stack

- `landing` on port `3000`
- `docs` on port `3002`
- `app-gateway` on port `3001`

`app-gateway` is the stable front door for the product app. When the app stack
is restarting, it serves a maintenance page instead of a connection failure.

Persistent VPS data lives on the host at:

```text
/opt/oneglanse/storage
```

Inside the containers, that path is mounted as:

```text
/storage
```

## Deploy

```bash
pnpm install
pnpm self-host
```

That is enough for a working self-hosted deployment.

Routine app-only redeploy:

```bash
docker compose up -d --build
```

Routine public-site redeploy:

```bash
docker compose -f docker-compose.public.yml up -d --build
```

If you want helper scripts instead:

```bash
pnpm self-host:app
pnpm self-host:public
```

## Public Ports

Default exposed ports:

- `3000` - landing
- `3001` - app gateway
- `3002` - docs
- `3333` - auth upload API

For production, place Nginx, Caddy, or another reverse proxy in front of these.

Typical routing:

- `domain.com` -> `127.0.0.1:3000`
- `app.domain.com` -> `127.0.0.1:3001`
- `docs.domain.com` -> `127.0.0.1:3002`

## Required VPS Variables

Most self-hosted deployments only need a small set of overrides in `.env`:

- `OPENAI_API_KEY`
- `BETTER_AUTH_SECRET`
- `APP_URL`
- `API_BASE_URL`
- `AGENT_AUTH_UPLOAD_TOKEN`

Optional:

- `GOOGLE_CLIENT_ID`
- `GOOGLE_CLIENT_SECRET`
- `PROXY_*`
- `THORDATA_PROXY_API_URL`

## Auth on VPS

The VPS never opens an interactive login browser.

When provider auth is missing on the VPS, use the same auth screen on your
local machine and upload the sessions to the VPS:

```bash
pnpm auth -- --upload-url http://YOUR_VPS_HOST:3333/auth/sessions --upload-token YOUR_TOKEN
```

That flow:

- captures auth locally
- stores the auth bundle locally
- uploads it to the VPS
- invalidates the matching runtime profiles on the VPS

The next provider run on the VPS reseeds from the uploaded auth bundle.

## Runtime State on VPS

The worker keeps two layers of state:

- auth bundles under `/storage/auth`
- persistent runtime profiles under `/storage/runtime`

Runtime behavior:

- if a runtime profile is missing, it is seeded from the auth bundle
- if the auth bundle changes, that profile is reseeded
- otherwise the persistent profile is reused directly

Google auth is shared by:

- `gemini`
- `ai-overview`

## Reverse Proxy and TLS

Expose only your reverse proxy publicly.

Recommended:

- bind public traffic to Nginx on `80/443`
- forward to `127.0.0.1:3000`, `3001`, and `3002`
- keep Redis, Postgres, and ClickHouse private

If you use Let’s Encrypt, terminate TLS at the reverse proxy.
